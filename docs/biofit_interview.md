## 1부. MSA 서비스의 회고

### 1.1 처음 만든 구조 — *"컨테이너 7개"라는 외형*

BioFit은 data_service, ai_service, group_service, feedback_service, db, vLLM, streamlit처럼 여러 컨테이너로 분리되어 있었지만, 모든 서비스가 같은 DB의 테이블에 직접 접근할 수 있었기 때문에 서비스 간 결합이 DB 스키마에 의존하는 형태였습니다. 예를 들어 ai_service가 data_service가 만든 테이블명을 하드코딩해서 읽는 구조에서는, data_service가 컬럼명이나 테이블 구조를 변경하면 ai_service가 런타임에 깨질 수 있습니다. 이는 MSA의 핵심인 독립 배포와 데이터 소유권 분리에 맞지 않습니다. 그래서 개선 과정에서는 서비스별로 소유 테이블을 나누고, 다른 서비스의 데이터가 필요할 때는 직접 DB를 조회하지 않고 HTTP API를 통해 요청하도록 변경했습니다. 이를 통해 결합 지점을 DB 스키마에서 API 계약으로 옮겼고, 각 서비스가 자신의 내부 데이터 구조를 더 독립적으로 변경할 수 있게 했습니다.




### 1.2 *왜* 그것이 문제였나 — 4 가지 구조적 결함

- 본 프로젝트의 분석 문서 §9.1에서 *"이 구조가 어떻게 깨지는가"*를 네 갈래로 진단했습니다.
- 첫째, *클라이언트 측에 의존하는 진행 상태*의 취약성이었습니다. 트레이너가 화면 새로고침·페이지 이동·세션 만료처럼 *클라이언트 상태를 잃는 모든 동작*을 하면 작업이 처음부터 다시 시작됐습니다. 원인은 *백엔드 마이크로서비스 측*(`ai_service`·`data_service`)에 *그 회원의 작업이 어디까지 진행됐는지를 박제하는 테이블과 그것을 조회할 endpoint*가 모두 부재했고, 그 정보가 Streamlit 세션 객체에만 박제됐기 때문입니다.
- 둘째, *한 서비스의 내부 변경이 다른 서비스의 런타임 깨짐으로 전파*되는 결합 패턴이었습니다. 어느 서비스가 자기 데이터 구조를 바꾸면, 그 구조에 의존하는 다른 서비스는 *런타임 첫 호출이 발생하기 전까지* 깨짐을 발견하지 못했습니다. 원인은 서비스 간 결합이 *기계가 강제하는 API 계약*이 아니라 *문자열로 hardcode된 테이블명·컬럼명*에 있었고, 컴파일러나 정적 분석 도구가 그 결합을 검증할 방법이 없었기 때문입니다.
- 셋째, *회원 수에 비례하여 테이블이 선형으로 늘어나는 저장 구조*였습니다. 회원이 한 명 늘 때마다 8~9개의 새 테이블이 생성되는 패턴이었습니다. 원인은 *데모 환경의 직관성*을 위해 회원별 동적 테이블(`{uid}_*`)을 생성하는 패턴을 선택했지만, 운영 환경의 표준은 *공통 테이블 + user_id 컬럼*의 정규화 구조라는 점을 초기 설계에서 충분히 의식하지 못했기 때문입니다.
- 넷째, *데이터의 의미·단위·형식 변경이 침묵 속에 흐르는* 결함이었습니다. 어느 컬럼의 단위나 범위가 변경되어도 *런타임 에러가 발생하지 않은 채* 그 변경된 데이터가 다음 단계로 흘러 *조용한 데이터 오염*을 만들었습니다. 원인은 데이터의 단위·범위·의미가 *코드 주석이나 PR 합의*에만 적혀 있고, Pydantic Field 제약처럼 *기계가 검증할 수 있는 계약*이 부재했기 때문입니다.
- 이 네 결함의 공통 메시지: **모놀리스의 단점(컴파일 타임 안전 부재)과 MSA의 단점(독립 배포·장애 격리 부재)을 *동시에* 가진 상태**. 즉 결합이 *없는 것이 아니라 잘못된 위치에 있었습니다*.

### 1.3 결과적인 MSA 구조 — *데이터 소유권 + 명시적 계약*

- 7개의 상시 컨테이너가 각자의 책임 영역과 *소유 테이블*을 가집니다.
  - **data_service** — Fitbit OAuth, raw 데이터 적재, 정규화 일별 features 산출 책임
    - `fitbit_raw` — Fitbit API에서 받아온 *원본 시계열*을 그대로 보관. 향후 features 정의가 바뀌어도 *원본 재가공*이 가능하도록 손대지 않은 형태로 유지
    - `fitbit_daily_features` — 모든 회원의 *일별 집계 features*가 wide-format으로 통합된 정규화 테이블. `(user_id, date)` 복합 PK라 회원이 늘어도 테이블 수는 일정. CatBoost 학습 입력의 1차 출처
    - `ingestion_runs` — 데이터 수집 작업의 진행 상태·실패 추적. *어느 회원의 어느 기간 적재가 성공/실패했는지* 추적용
  - **feedback_api** — 회원 자기보고 점수, 코칭 메시지 평가 적재 책임
    - `sleep_feedback` — 회원의 *수면 자기보고 점수*가 누적되는 테이블. PSQI C1 차원(주관적 수면의 질)의 입력 데이터
    - `predictions_feedback` — 코칭 메시지에 대한 *트레이너 thumbs up/down + 1~5 평점·코멘트*가 적재. predictions와 외래키로 연결되어 *어느 추론*이 *어떤 평가*를 받았는지 추적되는 *평가 루프의 핵심*
  - **ai_service** — CatBoost 회귀, SHAP 설명, vLLM 호출, 추론 결과 영구 박제 책임
    - `predictions` — *어떤 호출의 결과*를 영구 보존. 메시지 본문, holdout RMSE/MAE, 모델 버전, 프롬프트 해시, 추천 슬롯 JSON까지 박제되어 *재현성과 회귀 추적*의 단일 출처
    - `token_usage` — vLLM 호출의 *토큰 비용* 누적. 운영 비용 모니터링과 회원당 호출 한도 정책의 입력
    - `model_runs` — 추론 단계별 *상태 머신* (`queued → running → succeeded/failed`). *어느 단계에서 자주 깨지는가*를 SQL 한 줄로 분석할 수 있게 만든 분산 추적의 토대
    - `idempotency_log` — 멱등성 키별 *응답 캐시*. 같은 키로 재호출되면 vLLM·CatBoost 재실행 0건이라 *토큰 누수와 결과 중복*을 차단
  - **group_service** — 회원의 preferred_slots 기반 파트너·그룹 매칭 책임
    - `users` — 회원 메타정보(이름, 운동 레벨, 이미지 URL 등)
    - `preferred_slots` — 회원이 *운동 가능하다고 등록한 시간대*. 매칭의 1차 입력
    - `group_sessions` — 헬스장이 운영하는 *그룹 운동 세션* 정의(시작·종료 시각, 정원, 난이도)
    - `match_history` — 과거 매칭 결과 기록. *추천 → 만남*의 conversion 분석 입력
  - **streamlit / streamlit-feedback** — 트레이너·회원 UI 계층. *영속 데이터를 소유하지 않습니다*. 모든 상태는 백엔드 서비스에 위임되고, UI는 그 서비스의 HTTP endpoint를 호출만 합니다.
  - **db (PostgreSQL)** — *논리적 소유권 분리*가 강제된 공유 저장소. 위 테이블들이 *서비스별로 read·write 권한이 나뉘어* 있고, 다른 서비스는 *해당 서비스의 HTTP endpoint를 통해서만* 그 테이블에 접근할 수 있습니다.
- 핵심 원칙: *자기 소유 테이블에는 직접 access OK, 다른 서비스 소유 테이블에는 반드시 API 경유*. 이 원칙이 *서비스 간 결합 지점이 어디에 있는가*를 정의합니다.
- 서비스 간 통신은 *HTTP + Pydantic 모델 + OpenAPI 자동 명세*로 통일됐습니다. 입력·출력의 구조·단위·범위가 *Pydantic Field 제약*으로 명문화되어, 데이터 약속이 *기계가 강제하는 형태*로 박제됐습니다.
- 추론 호출의 진행 상태는 *`ai_service`의 상태 머신*(`model_runs` 테이블)에 박제되고, 같은 입력의 재호출은 `Idempotency-Key` 헤더로 *기존 결과를 즉시 반환*합니다. 진행 상태가 더 이상 클라이언트에만 의존하지 않습니다.


### 1.4 이 구조가 §1.2의 네 결함을 *어떻게* 해결하는가

- *결함 1 — 클라이언트 측 진행 상태 의존*은 **백엔드 서비스 측 상태 머신**으로 해결됩니다. 추론 단계별 상태(`queued → running → succeeded/failed`)가 *`ai_service`의 `model_runs` 테이블*에 박제되고, *데이터 수집 작업*도 `data_service`의 `ingestion_runs` 테이블에 박제됩니다. 클라이언트가 *어디까지 진행됐는지*를 *해당 서비스의 endpoint를 호출해 조회*할 수 있게 됐습니다. 또 `ai_service`의 `Idempotency-Key` 헤더로 재호출이 안전해져, 새로고침이 작업을 처음부터 다시 시작시키지 않습니다.
- *결함 2 — 한 서비스 내부 변경의 런타임 깨짐 전파*는 **명시적 HTTP 계약 + Pydantic 검증**으로 해결됩니다. 한 서비스가 다른 서비스의 데이터에 접근할 때 *문자열로 테이블명을 알지 않고* HTTP endpoint를 호출합니다. 데이터 구조가 변경되면 *Pydantic 모델 정의를 함께 갱신해야* 하므로 그 변경이 *PR diff에 명시적으로 노출*됩니다. *런타임 첫 호출까지 모르는 결합*이 *PR 리뷰 시점에 보이는 결합*으로 옮겨졌습니다.
- *결함 3 — 회원 수에 비례한 테이블 선형 증가*는 **공통 정규화 테이블 + user_id 컬럼**으로 해결됩니다. 모든 회원의 일별 데이터가 *데이터 종류당 하나의 테이블*에 적재되고, 회원이 늘어도 테이블 수는 일정합니다. 인덱스·백업·모니터링·보존 정책이 *한 번 정의되면 모든 회원에 일관 적용*되고, 회원 삭제도 `WHERE user_id = ?` 한 줄로 처리됩니다.
- *결함 4 — 데이터 의미·단위 변경의 silent corruption*은 **Pydantic Field 제약**으로 해결됩니다. 단위·범위가 *코드 측 제약*으로 명문화되어 변경 시 *런타임에 즉시 422 검증 오류*가 발생합니다. *조용히 흐르던* 단위 변경이 *호출 시점에 큰 소리로 깨지는* 변경이 됐습니다.
- 네 결함 모두 *결합을 없애지 않고 결합 지점을 옮긴* 결과로 해결됐습니다. 결합은 여전히 존재하지만 *기계가 검증할 수 있는 위치*, *변경 시점에 발견되는 위치*에 있습니다.

## 2부. ML 서비스의 회고

### 2.1 처음 만든 모델 — *efficiency 단일 회귀*

- 처음 만든 모델은 CatBoost 회귀로 *efficiency* 한 숫자를 예측하는 단일 타깃 구조였습니다.
- features는 22개 — 활동 시간대, 운동량, 수면 시간대, HRV/HR.
- LLM이 그 예측값과 SHAP 결과를 받아 한국어 코칭 메시지를 생성했습니다.
- 외형은 *AI 서비스*였지만,  *추천 로직의 근거*에서 결함이 있었습니다.

### 2.2 *왜* 그것이 문제였나 — 5가지 결함

 *efficiency만으로 수면 질을 대표*하는 한계였습니다. 이는 침대에 누워있는 시간 중 잠든 시간의 비율을 측정한 지표이기 때문입니다. 수면 시간, 수면 잠복기, 주관적 수면 질 을 포함하여 보다 타당한 수면의 질 예측으로 보완했습니다.


### 2.5 3차 보완 — *타깃 적합성, PSQI 4 차원 분리* (계획 단계)

- efficiency 단일 타깃이 *수면 질의 한 측면*만 본다는 한계를 정면으로 풀기 위해, 피츠버그 수면 질 지수(PSQI)의 임상 척도를 따라 4 차원으로 분해하기로 했습니다.
- 4 차원은 *주관적 수면의 질, 수면 잠복기, 수면 시간, 습관적 수면 효율*입니다. 각 차원은 0~3 점수로 매핑되고, 합산은 0~12입니다.
- 각 차원에 *별도 CatBoost 모델*을 학습합니다. 4 모델 분리는 *해석 우위*를 의도적으로 우선한 결정이었습니다 — 차원별 SHAP을 분리해 *어느 변수가 어느 차원에 영향*인지 분해 가능하게 만듭니다.
- 가장 중요한 *비교 프레임의 전환*은 다음과 같았습니다.
  - efficiency 모델과 PSQI 모델의 RMSE 직접 비교는 *axis가 달라 무의미*합니다.
  - PSQI는 efficiency를 *포함한* 더 큰 척도라 두 모델은 *대등한 비교 대상*이 아닙니다.
  - 따라서 *RMSE 우열*이 아닌 *information coverage*로 비교 프레임을 옮겼습니다 — efficiency 모델은 1 차원, PSQI 모델은 4 차원을 예측 가능.
- 4 차원 *신뢰도가 비대칭*임을 정직하게 박제했습니다. 잠복기는 Fitbit이 *침대에 누운 시각*을 정확히 못 줘 근사값이고, 주관은 feedback 입력률에 의존합니다. *신뢰도 가중 합산*은 후속 박제.

### 2.6 추천 로직의 진화 — *Forward Simulation의 함정과 해결*

- 처음 만든 추천 로직(Phase 2)은 *raw 데이터의 순서 통계*였습니다 — 효율 상위 일자의 시간대 분포 top-k.
- 사용자 측에서 자연스럽게 떠오른 발상은 *"학습된 모델에 다양한 시간대를 입력해서 가장 높은 효율을 예측하는 슬롯을 추천하면 어떨까"*였습니다 — *forward simulation*.
- 이 발상의 4가지 위험을 정직하게 짚어야 했습니다.
  - Tree 모델 특성상 모델은 학습 분포 안만 신뢰 가능하여 임의로 features를 바꾸면 *학습 분포 밖*으로 빠져 모델을 신뢰할 수 없게 됩니다. 즉 forward simulation을 *positivity 영역*(시도해 본 슬롯)으로 *제약*하면 위험을 감소할 수 있습니다.
- 이 통찰 위에 4 단계 추천 흐름을 설계했습니다 (Phase F).
  - *Cold-start 4주 exploration* — 신규 회원은 *주 단위로 다른 시간대*로 운동 권장. 데이터 균등 수집.
  - *Positivity 영역 정의* — 회원이 *3일 이상 시도해 본* 시간대만 후보.
  - *Realistic input* — 그 시간대에 운동한 *과거 day의 features 평균*을 base로. 한 변수만 바꾸지 않고 *전체 features 정합* 자동 보존.
  - *PSQI 합산 최소화* — 4 모델 모두에 시뮬레이션, 합산이 가장 낮은 슬롯 추천.
- 이 흐름의 가치는 *forward simulation을 안전한 영역에서만 사용*한다는 점이었습니다. *완전히 빼지 않고* 위험을 *제약*으로 풀었습니다.

### 2.7 트레이너 카드의 분리 — *정형/자연어*

- LLM이 *정형 데이터까지* 출력에 포함하면 환각 위험이 커집니다. 점수를 잘못 적거나, 표 형식이 깨지거나, 합산을 잘못 계산할 수 있습니다.
- 그래서 트레이너 카드를 두 영역으로 분리하기로 했습니다.
  - *정형 영역* (PSQI 점수, 추천 슬롯, 차원별 SHAP, 예측값) → HTML/Streamlit 직접 렌더. LLM 안 거침.
  - *자연어 영역* (회원에게 전할 한국어 한 단락) → LLM. 톤·표현·격려에는 LLM이 자연스러움.
- 이 분리는 분석 문서 §5.1의 *LLM 출력 검증* 부담도 줄였습니다. *자연어 한 단락만* 검증하면 되고, 정형은 *코드 측 책임*입니다.
- LLM 프롬프트에 *"숫자·점수·표는 출력하지 마세요"*를 명시하고, 출력 후 정규식으로 검증합니다. LLM이 지시를 무시할 위험에 대한 마지막 가드.
